## 서론
[Chat App](https://github.com/taeyoung-no/chat-app)에서 방 목록과 메시지를 실시간 업데이트하기 위해 Socket.IO를 사용하고 있습니다. 방 목록은 로그인 없이도 볼 수 있으며 방에 접속하거나 메시지를 전송하는 데는 로그인이 필요합니다. 인증은 session 방식을 사용합니다. 그리고 웹소켓에 연결하면 `sessionMiddleware`를 직접 호출해서 session을 저장합니다.   

```ts
io.use((socket, next) => {
  const req = socket.request as express.Request & { session: any }

  sessionMiddleware(req, {} as any, (err) => {
    if (err) return next(new Error("Session error"))

    if (req.session?.userId) {
      socket.data.userId = req.session.userId
      socket.data.username = req.session.username
    }
    next()
  })
})
```

그런데 로그인 후 방에 접속하려 하면 로그인하라는 에러 메시지가 출력됐습니다. 그래서 `req.session`을 출력해봤는데 `undefined`였습니다. 즉, 로그인을 해도 session 정보가 저장되지 않는 현상이 발생했습니다. 이걸 해결하기 위해 겪은 시행착오를 정리했습니다.

---

## 본론
### WebSocket Handshake
> the WebSocket handshake uses the HTTP Upgrade header to change from the HTTP protocol to the WebSocket protocol [Wikipedia](https://en.wikipedia.org/wiki/WebSocket)

`socket.request`는 웹소켓에 처음 연결할 때의 HTTP request일 것입니다. 즉, 로그인하기 전에 웹소켓에 연결했다면 로그인한 후에도 `socket.request`에 session 정보가 없는 것입니다.

### Namespace 도입
따라서 Socket.IO의 namespace를 도입했습니다. 로그인하기 전에는 인증 정보를 저장하지 않는 public 소켓, 로그인 후에는 인증 정보를 저장하는 private 소켓에 연결하는 겁니다. 로그인 후에 새로운 소켓에 연결하기 때문에 session 정보를 잘 저장하지 않을까요? route-based public/private namespace를 구현했습니다. 즉, 메인 페이지(`/`)에서는 public 소켓, 그 외 페이지에서는 private 소켓에 연결합니다.

### Manager
namespace를 도입했지만 로그인한 후에도 여전히 session 정보가 없는 것을 확인했습니다. Socket.IO의 동작에 대해 이해하지 못한 부분이 있는 것으로 판단, [공식 문서](https://socket.io/docs/v4/client-options/)를 통해 조사했습니다.

> A Socket instance is the interface which is used to sends events to — and receive events from — the server. It belongs to a given namespace.

namespace별로 소켓이 분리돼 있다는 내용인데 제가 이해하고 있던 것과 일치합니다.

> A Manager instance is in charge of the low-level connection to the server (established with HTTP long-polling or WebSocket). It handles the reconnection logic.

Manager에 대해 알게 됐습니다. 내부적으로 low-level 연결을 담당하는 instance가 존재했습니다.

> A single Manager can be attached to several Socket instances.

Manager 하나가 여러 소켓, 즉 여러 namespace를 담당할 수 있다는 것도 알게 됐습니다. public에 연결할 때 생성된 Manager를 private 연결할 때 재사용하기 때문에 여전히 인증 정보가 없는 것입니다. `forceNew` 옵션을 사용하면 새로운 Manager를 생성하여 웹소켓에 연결할 수 있습니다. 로그인 직후 해당 옵션 사용하여 private 소켓에 연결하면 정상적으로 session 정보를 가질 수 있습니다.

### 더 좋은 방법들
1. 브라우저에서 로그인 시 웹소켓 끊었다가 다시 연결하면 됩니다. `forceNew` 옵션 사용하는 것도 새로운 물리적인 연결을 강제하는 거니까 재연결하는 것과 오버헤드는 같거나 비슷할 것으로 예상합니다.

    사실 처음에 이 방법으로 해결했습니다. 나중에 부하 테스트를 위해 Artillery 라이브러리를 도입했습니다. 해당 라이브러리에는 웹소켓을 재연결하는 방법이 없고 namespace를 전환하는 방법은 있습니다. namespace 전환하면 완전히 새롭게 웹소켓에 연결하는 건 줄 알고 namespace를 도입했습니다. 결과적으로 namespace를 분리하기 위해 서버 코드는 더 복잡해졌고 클라이언트도 `forceNew` 옵션을 사용하게 됐습니다.

2. Server-Sent Event(SSE)를 도입하기 전이라서 방 목록을 실시간 업데이트하기 위해 웹소켓을 사용했습니다. 방 목록 업데이트를 위해 SSE를 사용하고 방 접속 및 메시지 전송에는 웹소켓을 사용하면 됩니다. 웹소켓 연결할 때 인증 정보 없으면 에러 메시지를 emit하여 private 소켓처럼 사용할 수 있습니다.

## 결론
WebSocket의 handshake 과정과 Socket.IO의 Manager에 대해 알게 됐습니다. 더 좋은 방법들이 있었지만 억지로 Socket.IO를 사용하며 겪은 시행착오는 기록할 만하다고 판단해서 작성했습니다.
