## 서론
[HTTP Server From Scratch](https://github.com/taeyoung-no/http_server) 프로젝트는 『윤성우의 열혈 TCP/IP 소켓 프로그래밍』의 예제 코드를 기반으로 진행했습니다. 교과서를 따라했다가 겪은 시행착오를 정리했습니다.

---

## 시행착오
### ERR_CONTENT_LENGTH_MISMATCH
`ERR_CONTENT_LENGTH_MISMATCH` 에러가 발생해서 코드를 리뷰했습니다. 교과서에서는 `Content-length:2048`으로 하드코딩했습니다. 실제 길이를 동적으로 계산하도록 수정하여 해결했습니다.

### send_error()에서 content를 사용하지 않는 현상
교과서의 `send_error()`에서는 `content`를 선언하고 사용하지 않습니다. 사용 안 해도 `웹 페이지를 찾을 수 없습니다` 따위가 출력됩니다. 하지만 저는 사용하도록 수정했습니다.

### content_type()의 예외 처리 능력 부족
교과서 예제 코드 기반으로 서버를 작성했더니 요청 받으면 `Segmentation Fault`에 의해 터지는 현상이 발생했습니다.

브라우저로 `GET` 요청을 한 번만 했는데 포트가 세 번 열립니다. 따라서 `GET /index1.html HTTP/1.1`를 제외한 다른 요청이 자동으로 전송되는 것으로 판단했습니다.

요청을 받을 때마다 `req_line`을 출력하게 했습니다. 결과적으로 `GET /.well-known/appspecific/com.chrome.devtools.json HTTP/1.1`가 출력됐습니다.

교과서의 `content_type()`은 `file_name`이 `name.ext` 형식이 아니면 `Segmentation Fault`가 발생합니다. 즉, `content_type()`의 인자로 정상적인 값만 들어올 것을 가정한 것입니다.

저는 `content_type()`의 인자로 비정상적인 값이 들어오면 `text/plain`을 반환하도록 예외 처리했습니다.

### request_handler()의 파싱 능력 부족
`content_type()`의 인자로 비정상적인 `file_name`을 전달하는 `request_handler()`가 `Segmentation Fault`의 근본적인 원인입니다. 즉, `request_hander()`가 `req_line`을 제대로 파싱하지 못하는 것입니다.

교과서에서는 `GET /index1.html HTTP/1.1`인 경우만 고려하여 `/` 이후부터 공백 이전까지를 `file_name`으로 파싱했습니다. 하지만 `GET /.well-known/appspecific/com.chrome.devtools.json HTTP/1.1`에 대해서는 `.well-known`을 `file_name`으로 파싱하게 됩니다. 저는 `strtok()` 대신 `strrchr()` 사용하여 적절하게 파싱하도록 수정했습니다.

## 결론
교과서가 오래되기도 했고 애플리케이션 레벨에서의 네트워크 프로그래밍을 어떻게 하는지 보여주기 위해서 간단하게 작성한 것 같습니다. 교과서에서 다루지 않는 문제들을 직접 겪고 해결했습니다.
