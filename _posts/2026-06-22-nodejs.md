## 서론
[Chat App](https://github.com/taeyoung-no/chat-app) 프로젝트 진행 중 예외 처리가 일관적이지 않고 서버와 클라이언트 여기저기 흩어져 있어서 예외 처리를 추가할 때마다 헷갈립니다. 그래서 새로운 기능을 만들거나 새로운 예외가 발견됐을 때 쉽게 확장할 수 있는 구조를 설계해봤습니다.

---
## 본론
### 응답 구조
응답은 `{ success, message, data }` 형식으로 통일했습니다. 성공이든 실패든 일관적인 구조로 응답이 와야 클라이언트에서 예외 처리할 때 편하다고 판단했습니다. 단, 클라이언트에서 `data`를 확인할 필요가 없는 경우엔 생략했습니다. `data`를 확인할 필요가 없는 경우는 다음과 같습니다.
1. 실패해서 보낼 데이터가 없는 경우
2. 로그아웃 등 성공해도 보낼 데이터가 없는 경우

### App Error
에러 클래스가 status code와 에러 메시지를 포함했으면 좋겠습니다. 그런데 JS 기본 제공 `Error` 클래스는 `message` 필드는 있지만 status code를 저장하는 필드가 없습니다. 그래서 `Error`를 상속받는 `AppError` 클래스를 작성했습니다. 필드에 `status`만 추가했습니다.
```ts
class AppError extends Error {
  status: number

  constructor(message: string, status: number = 500) {
    super(message)
    this.status = status
  }
}
```

### Error Handler Middleware
핸들러는 인자로 받은 에러에 `status`와 `message`가 있으면 그대로 담아서 응답하고 없으면 `status`는 500, `message`는 서버 문제라는 식으로 작성해서 전송합니다.
```ts
const errorHandler = (err: any, req: Request, res: Response, _: NextFunction) => {
  res.status(err.status || 500).json({ success: false, message: err.message || '서버 문제인 듯' })
}
```

### API에서
에러 원인을 특정할 수 있는 경우 `AppError`에 `status`와 `message`를 담아서 에러 핸들러를 호출합니다. DB 입출력 에러, 네트워크 에러 등 예상치 못한 에러인 경우 `try...catch`로 잡아서 호출합니다.

### 클라이언트에서
API 호출하는 부분을 service layer로 분리했습니다. 서버에서 status code로 4xx나 5xx를 보내면 Axios가 예외를 던집니다. Axios가 보낸 메시지는 `err.message`에, 서버가 보낸 메시지는 `err.response.data.message`에 담겨 있습니다. `catch`에서 예외를 잡으면 JS 내장 `Error`에 메시지를 담아서 다시 던집니다. 이때 서버가 보낸 메시지가 있으면 해당 메시지를, 없으면 Axios가 보낸 메시지를 담습니다. 이렇게 service layer를 작성하면 React Component에서는 예외가 있으면 잡아서 `err.message`를 확인하기만 하면 됩니다.
